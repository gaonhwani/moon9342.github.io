Introduction

이번에는 NoSQL DBMS의 대표격인 MongoDB에 대해서 알아보도록 하겠습니다.

MongoDB에 대한 내용을 알아보기에 앞서 기본적인 몇가지 용어와 개념부터 알고 넘어가도록 하겠습니다.

Big Data

Big Data는 이제는 아주 흔한 말이 됬습니다. 그런데 이 Big Data가 과연 무엇일까요? 그냥 단순히 큰 데이터 인가요?

쉽게 이해하자면 다음과 같이 Big Data를 말할 수 있습니다.

이전에는 분석이 불가능했지만, IT 기술이 발전하면서 다룰 수 있게 된 방대한 양의 데이터

예를 들자면, 휴대폰 통화내역, 실시간 교통량, SNS에서 실시간으로 발생되는 메시지, 인터넷 검색 내역 등등이 있을 수 있습니다.

Instagram이나 YouTube에서 하루에 발생하는 데이터량이 어마어마 합니다. 이런 데이터가 월단위, 연단위로 쌓이게 되면 물리적 이슈 그리고 그 데이터를 분석하기 위한 기술적인 이슈가 발생할 수 밖에 없습니다.

2019년 전 세계 데이터의 총 규모는 33 제타바이트 정도 입니다. 2025년에는 175 제타바이트까지 늘어날 전망이라고 합니다.

1 제타바이트(zettabyte) = 1000 엑사바이트(exabyte)

1 엑사바이트(exabyte) = 미 의회도서관 인쇄물의 10만배 정보량

상상할 수 없는 정도의 엄청난 데이터량 입니다.

아래의 그림은 인터넷 기업의 등장과 글로벌 디지털 데이터의 규모를 그래프로 그린 것입니다.

1995년에 이베이와 아마존이 설립되었고 1998년에 Google이 설립되었습니다. 빅데이터 관점에서 보면 이 회사가 시작점이 되었다고 볼 수 있습니다.

그 후에 2004~2006년까지 Facebook, Youtube, Twitter와 같은 서비스가 등장하면서 데이터량이 늘기 시작했습니다. 하지만 이때까지만 해도 그렇게 큰 데이터 량이 아니었습니다.

그러나 2007년 iPhone이 처음 등장하면서부터 데이터량이 기하급수적으로 증가하기 시작했습니다. 왜 그럴까요? 그 이전에는 웹에 대한 트레픽이 대부분 PC를 통해서 이루어졌습니다. 따라서 트레픽이 발생하는 시간이 대부분 업무시간에 집중되었죠. 하지만 iPhone을 기점으로 스마트폰이 대중화가 되었고 24시간 내내 트레픽이 발생하는 현상이 발생했기 때문에 데이터량이 엄청나게 많이 증가되기 시작했습니다.

이때부터 기존의 솔루션으로 이 데이터를 처리할 수 없다는 점은 인식하기 시작했고 여러 솔루션들이 등장해 이 문제를 해결해 나가기 시작했습니다. 아마존과 구글은 2000년대 초반에 이미 이런 문제들을 예상했고 그에 대한 솔루션들을 만들어 나가기 시작했습니다.

이번에는 빅데이터의 특징에 대해서 알아보도록 하죠. 빅데이터는 다음과 같은 특징을 가지고 있습니다.

  • 데이터의 양(Volume)
  • 데이터의 생성 속도(Velocity)
  • 데이터 형태의 다양성(Variety)

일반적으로 3V라고 불리는데 데이터의 양은 쉽게 알 수 있습니다. 데이터가 많다는 의미입니다. 그것보다 더 중요한 것은 그런 많은 데이터가 엄청난 속도로 쏟아져 나온다는 것입니다. 거기에 기존 솔루션은 정형화된 데이터를 다루는데 초점을 맞추고 있는데 빅 데이터는 그 데이터의 형태가 상당히 다양하다는 것입니다.

빅데이터는 이름이나 주민번호, 성별, 나이 이런 정형적인 데이터도 중요하게 여기지만 그 보다는

당신이 어디에서 무엇을 하며 어떤 방식을 선호하는가?

와 같은 비 정형적인 데이터를 로그를 이용해 기록하고 이를 더 중요하게 여기고 있습니다. 이를 분석해 더 나은 정보를 추출할 수 있기 때문입니다.

잠깐 기존 데이터와 빅데이터의 차이점을 표를 통해 살펴보도록 하죠.

여기서 한가지 주의해야 할 점은 이 두가지 시스템들은 서로 대응관계가 아니라는 점입니다. 각자의 시스템이 가지고 있는 단점을 서로 보완해주는 것이라고 이해하는게 맞습니다. 다시말하면 전통적인 RDBMS에서 NoSQL로 패러다임이 이동하는것이 아니고 둘 모두 상호 보완적인 관계를 통해 데이터를 더 잘 처리하자는 의미로 보시면 된다는 것입니다.

NoSQL의 특징

먼저 NoSQL이 등장하게 된 배경부터 알아보도록 하겠습니다.

기존 컴퓨팅 시스템은 다음과 같은 특징을 가지고 있습니다.

  • 기업의 업무를 자동화 하고 효율화 하는 목적
  • 기업의 복잡한 데이터를 저장하고 그 데이터 간의 관계를 정의하고 분석하는데 최적화
  • 기업의 업무 시스템은 해당 기업의 판매와 생산을 지원하기 위한 것
  • 생성되는 데이터량은 한계를 가지고 있음

하지만 2000년대에 들어서면서 세상이 바뀌기 시작했습니다.

  • 인터넷의 발전과 더불어 SNS 서비스의 활성화.
  • SNS 서비스는 전세계 사용자를 대상으로 한 서비스이기 때문에 대규모 데이터 생성
  • 이러한 데이터는 기존 기업 데이터에 비해 매우 단순한 형태를 가짐

즉, 기존의 시스템에서 사용하는 데이터는 복잡한대신 양이 적은 데이터 였다면 현재 생산되는 데이터는 대규모이고 단순화 되었다는 특징이 있습니다.

결국 기존의 데이터 저장 시스템으로는 커버할 수 없는 여러 가지 한계를 야기했고 결국에는 새로운 형태의 데이터 저장 기술을 요구하게 되었습니다.

그래서 단순 데이터를 가장 많이 보유하고 이에 대한 대용량 데이터 처리에 대한 요구가 가장 많았던 Google과 Amazon에 의해 Bigtable과 Dynamo라는 논문이 발표되었고 이 두 논문은 새로운 데이터 저장 기술을 만들어 내는 시발점이 되었습니다. 결국 기존 Oracle로 대변되던 RDBMS 중심의 데이터 저장 기술에서 새로운 데이터 저장 기술인 NoSQL이 등장하는 계기가 되었습니다.

NoSQL은 Not Only SQL의 약자로 기존 RDBMS의 형태의 관계형 데이터베이스가 아닌 다른 형태의 데이터 저장 기술을 의미합니다.

다시한번 말하지만 RDBMS와 NoSQL은 경쟁관계 혹은 대체관계가 아니라 상호보완관계라는 점을 인지해야 합니다.

다음은 주요한 NoSQL의 특징입니다.

NoSQL이라고 해서 RDBMS 제품군 처럼 공통된 형식의 데이터 저장방식(Table)과 접근방식(SQL)을 갖는 제품군이 아니라 RDBMS와 다른 형태의 데이터 저장 구조를 총칭하는 의미이며 제품에 따라 그 특성이 매우 달라서 NoSQL을 하나의 제품군으로 정의할 수 없습니다.

RDBMS가 데이터의 관계를 Foreign Key등으로 정의하고 이를 활용해 Join등의 관계형 연산을 하지만 NoSQL은 데이터 간의 관계를 정의하지 않습니다.

RDBMS의 복잡도와 용량 한계를 극복하기 위한 목적으로 등장한 만큼, 페타 바이트급의 대용량 데이터를 저장할 수 있습니다.

NoSQL은 기존의 RDBMS처럼 하나의 고성능 머신에 데이터를 저장하는 것이 아니라 일반적인 서버 수십, 수백대를 연결해 데이터를 저장 및 처리하는 구조를 가집니다.

분산형 구조를 통해 데이터를 여러 대의 서버에 분산해 저장하기 때문에, 데이터의 분실과 장애발생에 상대적으로 견고한 특성을 가집니다.

NoSQL은 RDBMS와는 다르게 테이블의 스키마가 유동적입니다.

NoSQL의 종류

NoSQL은 저장방식에 따라서 몇가지 종류로 나뉘어집니다.

  • Key/Value Store

대부분의 NoSQL은 Key/Value의 개념을 가지고 있습니다. Unique한 key에 하나의 value를 가지고 있는 형태이고 put(key,value), get(key) 형태의 API를 사용합니다. 가장 대표적인 솔루션으로는 Redis가 있습니다.

  • Oredered Key/Value Store

다른말로 BigTable-style Database라고 합니다. 데이터가 내부적으로 Key를 순서로 Sorting 되어 저장되는 구조입니다. 아래 그림처럼 Key안에 (column:value) 조합으로 된 여러개의 필드를 가지는 구조입니다. 단순 Key/Value Store에서 조금 더 발전된 형태라고 볼 수 있습니다. 대표적인 솔루션으로 Apache Hbase, Apache Cassandra가 있습니다.

  • Document Key/Value Store

다른말로 Document Database라고 합니다. Key/Value Store의 확장된 가장 진보한 형태입니다. 저장되는 Value의 데이터 타입으로 Document라는 구조화된 데이터 타입(JSON, XML, YAML)을 사용합니다. 복잡한 계층 구조의 표현이 가능하며 제품에 따라서 추가적인 기능(Sorting, Join)지원이 되고 있습니다. 대표적인 솔루션으로 MongoDB가 있습니다.

NoSQL의 장단점

먼저 Modeling 관점에서 Relational Data Modeling과 NoSQL Data Modeling의 특징부터 알아보기로 하겠습니다.

  • Relational Data Modeling

전형적으로 가용한 데이터 구조에 기반하고 있습니다. 즉, 내가 가지고 있는 데이터가 무엇인가? 라는 점이 가장 중요합니다. 내가 사용할 수 있는 도구는 정해져있기 때문에 사용자의 모든 요구를 반영하기는 쉽지 않은 구조입니다.

그래서 데이터 모델을 먼저 정의한 후 어플리케이션에 맞는 쿼리를 개발하는 식으로 진행

저장하고자하는 도메인 모델 분석 -> 개체간의 관계(relation) 식별 -> 테이블 추출 -> 테이블을 이용한 쿼리 구현 순으로 진행합니다.

  • NoSQL Data Modeling

어플리케이션의 특징적인 데이터 접근 패턴에 따라 모델링을 합니다. 쉽게 말하면 고객의 요구사항이 무엇인가에 더 집중하는 방식입니다. 기존 관계형은 내가 가지고 있는 답(데이터)에 근거해서 고객의 요구사항을 받아들이는 구조였는데 NoSQL은 고객이 원하는 답이 무엇인지를 먼저 파악하고 그 답을 제공해 줄 수 있는 데이터는 어떻게 모델링을 해야 할지를 고민하고 결정해야 합니다.

어플리케이션의 필요한 쿼리와 성능을 먼저 정의한 후 요구사항에 부합되도록 데이터 모델을 구성

도메인 모델 분석 -> 쿼리 결과 도출 -> 테이블(데이터 저장 모델) 설계 순으로 진행합니다.

따라서, NoSQL 모델링은 Relational 모델링보다 더 깊은 데이터 구조 및 알고리즘에 대한 이해가 필요하고 이에 따른 물리적인(디스크)인 디자인도 같이 병행해야 합니다. 대부분의 NoSQL 제품들은 보안에 취약하기 때문에 방화벽과 같은 별도의 보안체계를 마련해야 합니다.

이번에는 RDBMS의 특징과 NoSQL의 특징을 간단하게 비교해서 보기로 하죠.

  • RDBMS의 특징

범용적이고 고성능

대부분의 경우 관계형 데이터베이스를 사용하는 것이 안정적

Transaction으로 데이터의 일관성을 보장받고 데이터의 불일치를 방지할 수 있다.

정규화를 전제로 하고 있기 때문에 업데이트 시 비용이 적게 든다

데이터베이스 설계 당시부터 불필요한 중복이 삭제된다.

복잡한 형태의 쿼리도 가능하며 이미 성숙한 기술이다.

대량의 데이터 입력 처리에는 한계를 가진다.

테이블의 인덱스 생성이나 스키마를 변경할 때 많은 오버헤드가 발생한다.

개발과 운영시 컬럼을 확정짓기 어려운 경우 사용하기 쉽지 않다.

  • NoSQL의 특징

NoSQL은 범용성 보다는 특정 용도로 특화되어 있기 때문에 각 NoSQL 솔루션의 특징을 알 필요가 있다.

기본적으로 NoSQL의 Join 연산은 대부분 불가능.

데이터를 여러 서버에 분산시키기에 유리하며 배열 형식의 데이터를 빠르게 처리하고 많은 데이터를 저장할 때 유리
